# 07. MVP, этапы и критерии приемки

## Состав MVP

MVP должен позволять администратору:

- создать объект;
- задать бюджет;
- принять или отклонить заявку бригадира;
- назначить бригадира на объект;
- добавить расход по любому активному объекту;
- редактировать и мягко удалять любые неудаленные расходы;
- смотреть расходы объекта;
- смотреть расходы бригадира;
- выгрузить CSV;
- архивировать и восстановить объект.

MVP должен позволять бригадиру:

- подать заявку на доступ, если он еще не зарегистрирован;
- войти по Telegram ID после принятия заявки;
- увидеть свои активные объекты;
- добавить расход одного из четырех типов;
- редактировать и мягко удалить свои расходы по активным объектам;
- посмотреть свои расходы;
- посмотреть все расходы назначенного объекта;
- увидеть бюджет, расходы и остаток по объекту.

## Этапы реализации

### Этап 1. Каркас проекта

- Node.js-приложение;
- конфигурация;
- подключение Telegram-бота;
- подключение базы данных;
- миграции;
- базовый health check, если будет HTTP-сервер.

### Этап 2. Пользователи и авторизация

- таблица пользователей;
- таблица заявок бригадиров;
- bootstrap администратора через `ADMIN_TELEGRAM_IDS`;
- middleware авторизации;
- подача заявки неизвестным пользователем через `/start`;
- принятие или отклонение заявки администратором;
- меню по ролям;
- отказ неизвестным пользователям вне сценария заявки.

### Этап 3. Объекты и бюджет

- CRUD объектов для администратора;
- бюджет объекта;
- расчет расходов и остатка;
- архивирование и восстановление.

### Этап 4. Бригадиры и назначения

- управление бригадирами после принятия заявки;
- блокировка;
- мягкое удаление;
- назначения на объекты;
- фильтрация объектов для бригадира.

### Этап 5. Расходы

- добавление четырех типов расходов;
- редактирование и мягкое удаление расходов по матрице прав;
- валидации;
- сохранение фото через Telegram `file_id`;
- просмотр расходов.

### Этап 6. Отчеты и CSV

- отчет по объекту;
- CSV-экспорт.

## Критерии приемки

### Авторизация

- неизвестный Telegram ID может подать заявку через `/start`;
- неизвестный Telegram ID вне сценария заявки не получает доступ к функциям бота;
- пользователь с pending-заявкой видит только ожидание решения администратора;
- заблокированный пользователь получает отказ;
- пользователь из `ADMIN_TELEGRAM_IDS` получает роль администратора;
- администратор видит административное меню;
- бригадир видит меню бригадира.

### Объекты

- администратор может создать объект с бюджетом;
- объект отображает бюджет, расходы и остаток;
- архивный объект скрыт у бригадира;
- по архивному объекту нельзя добавить расход.

### Бригадиры

- администратор может принять заявку и создать бригадира;
- администратор может отклонить заявку без создания бригадира;
- администратор может назначить бригадира на объект;
- бригадир видит только назначенные активные объекты;
- заблокированный бригадир не может пользоваться ботом.

### Расходы

- ручная закупка рассчитывает сумму автоматически;
- чек/накладная сохраняет сумму и фото, если фото передано;
- безналичная операция сохраняет сумму;
- оплата услуг сохраняет сумму;
- каждый расход влияет на остаток бюджета;
- расход сразу учитывается в бюджете без подтверждения администратором;
- превышение бюджета не блокирует расход и показывает отрицательный остаток;
- администратор может редактировать любой неудаленный расход;
- администратор может мягко удалить любой неудаленный расход;
- бригадир может редактировать и мягко удалить только свой расход по активному объекту;
- бригадир не может редактировать или удалить чужой расход;
- бригадир видит все расходы назначенного объекта;
- фото чека/накладной сохраняется как Telegram `file_id`;
- бот может повторно отправить фото по сохраненному Telegram `file_id`;
- удаленные расходы не влияют на остаток.

### Отчеты

- администратор может получить отчет по объекту;
- CSV содержит требуемые колонки;
- CSV формируется одним файлом из карточки выбранного объекта;
- CSV доступен в разрезе объекта;
- CSV формируется без фильтра по датам;
- CSV не содержит мягко удаленные расходы.
