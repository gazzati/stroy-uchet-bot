# 09. Техническая архитектура

## Стек реализации

MVP реализуется на Node.js.

Базовые решения:

- язык выполнения: Node.js;
- язык разработки: TypeScript;
- основной интерфейс: Telegram Bot API;
- Telegram-библиотека: grammY;
- состояние диалогов: grammY `session` (in-memory storage в MVP);
- база данных: PostgreSQL;
- query builder: Kysely;
- тестовый фреймворк: Vitest;
- пакетный менеджер: Yarn;
- запуск бота в MVP: polling;
- хранение фото документов: Telegram `file_id`;
- денежные значения: целые копейки;
- язык интерфейса: русский;
- деплой: Docker image + GitHub Actions + Docker Compose на VPS.

## Рекомендуемая структура приложения

Предварительная модульная структура:

```text
src/
  config/
  db/
  domain/
  services/
  telegram/
  reports/
  utils/
```

Назначение модулей:

- `config` - переменные окружения, включая `ADMIN_TELEGRAM_IDS` и токен Telegram-бота;
- `db` - подключение к БД, миграции, транзакции;
- `domain` - доменные типы, бизнес-правила, проверки прав;
- `services` - прикладные операции: объекты, бригадиры, расходы, отчеты;
- `telegram` - команда `/start`, inline-меню, callback-роутинг и пошаговые wizard-сценарии ввода;
- `reports` - формирование отчетов и CSV;
- `utils` - форматирование денег, дат, CSV и общие помощники.

## Архитектурные правила MVP

- Бизнес-правила не должны жить только в Telegram-обработчиках.
- Проверка прав выполняется перед каждым действием, меняющим или раскрывающим данные.
- Все операции создания, редактирования и удаления расходов должны проходить через сервисный слой.
- Расчет бюджета и остатка должен использовать только неудаленные расходы.
- Мягкое удаление расходов и назначений выполняется через `deleted_at`.
- Архивирование объекта выполняется через `status = 'archived'` и `archived_at`.
- Фото документов не скачиваются в свое хранилище в MVP.
- Интерфейс построен на inline-кнопках (`callback_query`); единственная slash-команда — `/start`.
- Многошаговый ввод реализован через session-FSM (дискриминированный union состояний в `session`);
  черновики хранятся только в памяти процесса и теряются при рестарте — это допустимо, так как
  запись в БД происходит только после экрана подтверждения, а осиротевший после рестарта callback
  graceful-деградирует в показ главного меню.
- Плагины `@grammyjs/conversations` и `@grammyjs/menu` рассмотрены и сознательно не используются:
  `conversations` переигрывает диалог на каждый апдейт и требует оборачивать сайд-эффекты, а `menu`
  негибок для пагинации произвольных списков; вместо них — собственная FSM и ручная сборка `InlineKeyboard`.

## Конфигурация

Минимальные переменные окружения:

| Переменная | Назначение |
| --- | --- |
| `BOT_TOKEN` | Токен Telegram-бота |
| `DATABASE_URL` | Строка подключения к PostgreSQL |
| `ADMIN_TELEGRAM_IDS` | Список Telegram ID первых администраторов |
| `TZ` | Часовой пояс сервиса |

`ADMIN_TELEGRAM_IDS` содержит один или несколько Telegram ID, разделенных запятыми.

## Деплой

Деплой MVP выполняется по схеме, аналогичной `gazzati/fidar-vpn`:

- GitHub Actions запускает CI;
- после успешного CI на основной ветке собирается Docker image;
- image публикуется в Docker Hub с тегами:
  - `sha-<commit>`;
  - `latest`;
- VPS не делает `git pull` и не хранит исходный код приложения;
- на VPS лежит runtime-директория с `docker-compose.yml`, `.env`, `.env.deploy` и архивом логов;
- deploy job подключается к VPS по SSH;
- deploy job записывает текущий тег в `.env.deploy`;
- перед перезапуском текущие container logs архивируются в `logs/archive`;
- `docker compose` подтягивает нужный image tag и пересоздает сервис.

Предварительная структура runtime-директории на VPS:

```text
/home/stroy-uchet-bot/
  docker-compose.yml
  .env
  .env.deploy
  logs/
    archive/
```

Минимальные GitHub Secrets:

| Secret | Назначение |
| --- | --- |
| `DOCKERHUB_USERNAME` | Логин Docker Hub |
| `DOCKERHUB_TOKEN` | Токен Docker Hub |
| `SSH_HOST` | Хост VPS |
| `SSH_USER` | Пользователь VPS |
| `SSH_PRIVATE_KEY` | SSH-ключ для деплоя |

Rollback выполняется сменой `IMAGE_TAG` в `.env.deploy` на предыдущий `sha-<commit>` и перезапуском `docker compose`.

## Точки уточнения перед кодом

Перед началом реализации нужно выбрать:

- имя Docker Hub image;
- имя VPS runtime-директории.
