# 08. Принятые решения

Статус решений:

- `MVP` - фиксируем как правило первой версии.
- `После MVP` - не блокирует первую версию.
- `Подтверждено` - решение согласовано и отражено в спецификации.

## Решения для MVP

| Вопрос | Решение | Обоснование |
| --- | --- | --- |
| Как создать первого администратора? | Через переменную окружения `ADMIN_TELEGRAM_IDS` со списком Telegram ID администраторов при старте приложения. | Просто для запуска MVP и совпадает с исходным ТЗ: администраторы задаются на уровне БД/конфигурации. |
| Может ли один бригадир быть на нескольких объектах? | Да, подтверждено для MVP. | В строительных задачах это нормальный сценарий, модель связи многие-ко-многим гибче. |
| Может ли объект иметь несколько бригадиров? | Да, подтверждено для MVP. | Позволяет вести объект несколькими ответственными без переделки схемы. |
| Может ли администратор добавлять расходы? | Да, по любому активному объекту. Подтверждено для MVP. | Администратор должен иметь возможность внести расход вручную или исправить управленческую ситуацию без входа под бригадиром. |
| Может ли бригадир видеть расходы других бригадиров по тому же объекту? | Да, если расходы относятся к назначенному ему объекту. Подтверждено для MVP. | ТЗ говорит, что бригадир видит текущие расходы и остаток бюджета по своим объектам. Иначе остаток бюджета будет непрозрачным. |
| Кто может редактировать расходы? | Администратор - любые неудаленные расходы. Бригадир - только свои расходы по активным объектам. Подтверждено для MVP. | Сохраняет контроль администратора и дает бригадиру исправлять собственные ошибки. |
| Кто может удалять расходы? | Администратор - любые расходы. Бригадир - только свои расходы по активным объектам. Подтверждено для MVP. | Удаление мягкое через `deleted_at`, история не теряется. |
| Нужно ли ограничение на редактирование или удаление по времени? | В MVP нет. Бригадир может редактировать и мягко удалять свои расходы без ограничения по времени, пока объект активен. Подтверждено для MVP. | Меньше логики и меньше спорных сценариев. При необходимости добавим после запуска. |
| Нужна ли история изменения бюджета? | В MVP нет отдельного журнала. Храним только текущий бюджет и `updated_at`. Подтверждено для MVP. | Для MVP достаточно контроля текущего состояния. Историю можно добавить отдельной таблицей позже. |
| Нужно ли хранить историю назначений после открепления? | Да, через мягкое удаление связи `deleted_at`. Подтверждено для MVP. | Это помогает понимать, кто раньше был назначен на объект. |
| Нужно ли подтверждение расходов администратором? | Не входит в MVP. Расход сразу учитывается в бюджете после создания. Подтверждено для MVP. | В исходном ТЗ подтверждение указано в плане развития. |
| Что делать при превышении бюджета? | Разрешать расход, показывать отрицательный остаток и визуальное предупреждение. Подтверждено для MVP. | Запрет может мешать реальной стройке, где перерасход часто фиксируют постфактум. |
| Нужны ли копейки в пользовательском вводе? | Да. Ввод разрешает рубли и копейки, хранение всегда в копейках. Подтверждено для MVP. | Удобно для пользователя и корректно для расчетов. |
| Валюта | Только рубли. Подтверждено для MVP. | Другие валюты не заявлены в ТЗ. |
| Единицы измерения для ручной закупки | Свободный ввод с быстрыми подсказками: `шт`, `м`, `м2`, `м3`, `кг`, `л`, `уп`, `меш`. Подтверждено для MVP. | Не блокирует нестандартные материалы, но ускоряет частые вводы. |
| Комментарий обязателен? | Нет, комментарий опционален для всех типов расходов. Подтверждено для MVP. | ТЗ не требует обязательности, MVP должен быть быстрым для полевого ввода. |
| Хранить ли Telegram `file_unique_id`? | Нет. В MVP храним только Telegram `file_id`. Подтверждено для MVP. | `file_id` достаточно, чтобы бот мог повторно отправить фото документа. Дедупликация файлов в MVP не требуется. |
| Скачивать ли фото документов в свое хранилище? | В MVP нет. Храним только Telegram `file_id`; просмотр выполняется через повторную отправку фото ботом в Telegram-чат. Подтверждено для MVP. | Уменьшает инфраструктуру MVP. Собственное хранилище можно добавить позже. |
| Нужны ли фильтры отчетов по датам? | В MVP нет, но структура отчетов должна позволять добавить фильтры. Подтверждено для MVP. | Фильтрация по датам указана в плане развития. |
| Нужна ли пагинация CSV-экспорта? | Для MVP нет. Экспорт формируется одним файлом по выбранному отчету. Подтверждено для MVP. | Объемы на старте ожидаемо небольшие. Если данных станет много, добавим фильтры по датам и ограничения экспорта. |
| Нужна ли локализация? | Только русский язык. Подтверждено для MVP. | Целевая аудитория и ТЗ на русском. |
| Как деплоить сервис? | Через Docker image, GitHub Actions, Docker Hub и Docker Compose на VPS, по схеме `gazzati/fidar-vpn`. Подтверждено для MVP. | Позволяет деплоить immutable image по SHA без `git pull` на сервере и делать простой rollback через `IMAGE_TAG`. |
| Какие технологии использовать в Node.js-приложении? | TypeScript, grammY, Kysely, PostgreSQL, Vitest, Yarn, polling. | Это дает типизированный сервисный слой, простой Telegram polling для MVP и контролируемую работу с SQL. |

## Решения после MVP

| Вопрос | Решение |
| --- | --- |
| История изменения бюджета | Добавить таблицу аудита бюджета, если потребуется контроль изменений. |
| Подтверждение расходов | Добавить статусы расходов: `draft`, `pending`, `approved`, `rejected` или упрощенный вариант. |
| Уведомления о превышении бюджета | Добавить после базового учета и отчетов. |
| Собственное хранение фото | Добавить S3-совместимое хранилище, если Telegram `file_id` будет недостаточен. |
| Фильтры отчетов по датам | Добавить в отчетах и CSV после запуска MVP. |
| Excel-экспорт | Добавить после CSV. |

## Статус подтверждения

Все рабочие решения из этого раздела подтверждены владельцем продукта для MVP.
