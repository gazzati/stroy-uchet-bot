## Принятые решения MVP

Эти решения подтверждены и используются как правила первой версии.

- Один бригадир может быть назначен на несколько объектов.
- Один объект может иметь несколько бригадиров.
- Все суммы хранятся в копейках как целые числа.
- Валюта MVP - рубли.
- Фото документов в MVP хранятся через Telegram `file_id`.
- Расходы удаляются мягко через `deleted_at`.
- Архивный объект не принимает новые расходы.
- Администратор видит архивные объекты в отдельном разделе.
- Подтверждение расходов не входит в MVP.
- Администратор создается через `ADMIN_TELEGRAM_IDS`.
- Бригадиры подключаются через заявку в боте: `/start`, ввод имени и фамилии, затем решение администратора.
- Администратор может добавлять расходы по активным объектам, а редактировать и мягко удалять любые неудаленные расходы.
- Бригадир видит все расходы по назначенному объекту.
- Бригадир может редактировать и мягко удалять только свои расходы по активным объектам.
- Превышение бюджета не блокирует добавление расхода.
- Отчеты в MVP без фильтров по датам.
- Реализация сервиса - Node.js + TypeScript.
- Telegram-библиотека - grammY.
- Query builder - Kysely.
- Тестовый фреймворк - Vitest.
- Запуск бота в MVP - polling.
- Деплой - по схеме Docker Compose + GitHub Actions, аналогично `gazzati/fidar-vpn`.

## Следующий уровень детализации

Перед началом реализации спецификацию можно дополнить:

- финальными миграциями БД;
- точными текстами сообщений бота;
- схемами состояний FSM;
- правилами форматирования денежных сумм;
- детализацией модулей и интерфейсов внутри Node.js-приложения.
